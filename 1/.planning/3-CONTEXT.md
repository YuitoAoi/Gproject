# 阶段 3 上下文: 模型训练任务编排

本文件概述了作为第三阶段核心的模型训练任务编排模块的关键架构与实现决策。

## 1. 训练参数呈现：“精选与分层”模式

- **决策**: 前端界面将采用“精选与分层”的设计模式来展示 LLaMA-Factory 的训练参数。
- **理由**: 直接暴露所有参数会使用户不堪重负，且违背了“简洁直观”的设计原则。此模式能在易用性和功能强大之间取得平衡。
- **实现**: 
  1.  **默认视图**: 仅展示最核心、最常用的参数（如：基础模型、数据集、学习率、Epochs、LoRA 设置等），并为它们提供合理的默认值和解释性提示。
  2.  **高级设置**: 将不常用或更复杂的参数收纳在一个默认折叠的“高级设置”区域中，供需要深度定制的高级用户使用。

## 2. 任务执行机制：作为库导入并以编程方式调用

- **决策**: 训练任务将通过在 Celery worker 中**将 LLaMA-Factory 作为库导入并以编程方式调用**其核心训练函数来执行，而不是通过 `subprocess` 调用其命令行脚本。
- **理由**: 这是为了实现对训练任务生命周期的精确控制，特别是为了满足**安全中止**和**可靠的资源（GPU显存）释放**这两个严格要求。与 `subprocess` 的“黑盒”调用相比，此方法提供了“白盒”控制能力。
- **工作流**: 
  1.  Celery worker 接收到一个训练任务。
  2.  在 worker 进程内部，直接调用 LLaMA-Factory 提供的 Python API（例如，一个 `run_training` 函数），并传入所有参数。
  3.  训练逻辑在 worker 进程的直接控制下运行。

## 3. 任务生命周期管理：API 与中止机制

- **决策**: 设计一套 RESTful API 来管理训练任务的整个生命周期，并使用数据库作为状态的“唯一真实来源”，结合 Redis 作为信令通道。
- **理由**: 此方案确保了系统状态的一致性和健壮性，并提供了一种实现优雅中止的机制。
- **核心端点与工作流**: 
  1.  `POST /api/v1/training-tasks`: 
      - **功能**: 创建并启动一个新的训练任务。
      - **流程**: 后端在 `training_tasks` 数据库表中创建一条状态为 `Pending` 的记录，然后将包含该记录 ID 的作业分派给 Celery。
  2.  `GET /api/v1/training-tasks`: 
      - **功能**: 获取所有训练任务的列表及其当前状态。
  3.  `POST /api/v1/training-tasks/{task_id}/cancel`: 
      - **功能**: 请求中止一个正在运行的任务。
      - **流程**: 后端在此 API 被调用时，在 Redis 中设置一个“中止标志位”（例如，键 `cancel-task:{task_id}`）。
- **安全中止实现**: 
  - 在 LLaMA-Factory 的训练循环代码中，将植入一个检查点，该检查点会定期检查 Redis 中是否存在对应的“中止标志位”。
  - 如果检测到标志位，训练循环将**主动、安全地**中断，执行必要的清理代码（如 `torch.cuda.empty_cache()`），然后将数据库中的任务状态更新为 `Cancelled`。

这些决策共同确保了模型训练模块将是高度可控、可靠且用户友好的，能够满足生产环境的要求。

## 4. 模型保存、注册与版本控制

- **决策**: 训练完成后的模型权重需要被持久化保存，并注册到系统中，以供后续使用。
- **理由**: 这是连接模型训练与模型使用的关键环节，确保了训练成果的可用性和可追溯性。
- **实现**:
  1.  **保存策略**:
      - **最终模型**: 当一个训练任务成功完成时，其最终产出的模型权重（如 LoRA 权重）将被保存到一个统一、结构化的模型存储路径中。
      - **检查点 (Checkpointing)**: 在训练配置中，将提供选项以允许用户设置“每 N 个 Epochs 保存一次检查点”。这将把训练过程中的临时权重保存下来，用于支持断点续训功能。
  2.  **模型注册**:
      - **数据库记录**: 训练成功后，将在一个新的 `trained_models` 表中创建一条记录。该记录将包含：
          - 模型名称（可由用户指定或自动生成）。
          - 所基于的基础模型 ID。
          - 所使用的 `training_task` ID （用于追溯训练参数和数据集）。
          - 权重文件的存储路径。
          - 创建时间等元信息。
      - **版本控制**: 通过这种方式，我们可以对同一个基础模型进行多次微调，并保留所有版本的记录和权重，实现了模型的版本化管理。
  3.  **模型发现**: 其他模块（如“模型推理”）将通过查询 `trained_models` 表来动态地发现和加载可用的微调模型。
