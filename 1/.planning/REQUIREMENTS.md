# 项目需求文档 (REQUIREMENTS.md)

## 项目概述
- **项目名称**: 基于 LLaMA-Factory 的轻量级本地化大模型微调工作站
- **定位**: 一款面向本地/私有云环境的、集数据处理、模型微调、任务监控、模型推理于一体的“一站式”LLMOps 轻量级工作站。
- **主要目标**: 提供数据管理、模型训练任务编排、实时监控、模型推理与服务、全链路可观测性等功能，支持本地或私有云部署，确保轻量级且易用。

## 技术栈
- **核心 LLM 框架**: LLaMA-Factory
- **后端服务**: Python 3.10+
- **Web 框架**: FastAPI
- **任务队列**: Celery + Redis
- **ORM**: SQLAlchemy
- **数据库**: MySQL
- **消息缓存**：Redis
- **GPU 监控库**: pynvml
- **前端框架**: Vue 3
- **前端构建工具**: Vite
- **UI 组件库**: Element Plus
- **状态管理**: Pinia
- **图表库**: ECharts
- **实时通信**: WebSocket

## 前端界面设计原则

### 设计原则与目标

- **简洁直观 (Simplicity & Intuitiveness)**: 界面元素清晰，布局合理，用户无需复杂培训即可快速上手。
- **功能完整 (Completeness)**: 覆盖数据管理、模型训练、任务监控、模型推理等核心业务流程。
- **企业级风格 (Enterprise-Grade Aesthetics)**: 采用专业、稳重、现代的视觉风格，符合企业应用场景。
- **高效交互 (Efficient Interaction)**: 优化用户操作路径，提供实时反馈，减少用户等待时间。
- **响应式设计 (Responsive Design)**: 页面应能在不同屏幕尺寸（桌面端为主，考虑 tablet 适配）下提供良好的浏览和操作体验。
- **可扩展性 (Scalability)**: 页面结构和组件设计应便于未来功能的添加和迭代。

### 视觉与设计规范

- **色调**:
  - **主色调**: 采用中性、专业的蓝色或灰色系，如 `#3498db` (主蓝) 或 `#5A677A` (主灰)。
  - **辅助色**: 用于强调、警示或选中状态，如浅绿色 (`#2ecc71`) 用于成功提示，橙色 (`#f39c12`) 用于警告，红色 (`#e74c3c`) 用于错误或删除操作。
  - **背景色**: 通常为浅灰色 (`#f8f9fa`) 或白色 (`#ffffff`)，保持页面干净。
- **字体**:
  - 选用清晰易读的无衬线字体，如 Inter, Roboto, Helvetica Neue, 或系统默认字体。
  - 字号层级分明，例如：标题 (24px-32px), 副标题 (18px-22px), 正文 (14px-16px), 控件标签 (12px-14px)。
- **布局**:
  - 页面内部元素对齐规范，留白充足，避免拥挤感。
- **图标**:
  - 选用一套风格统一、语义明确的图标库（如 Font Awesome, Material Icons, Ant Design Icons）。
- **控件**:
  - 按钮（Primary, Secondary, Danger）、输入框、下拉选择器、复选框、单选框、开关（Toggle）、进度条、加载动画等，应遵循统一的设计规范。

## 功能模块需求

### 仪表盘 (Dashboard)
- **目的**: 提供项目整体状态的概览。
- **UI 元素**:
  - **关键指标卡片**: 如：总数据集数量、正在进行的训练任务数、可用 GPU 数量/状态概览。
  - **近期活动概览**: 如：最近完成的任务、最近上传的数据集。
  - **系统状态指示灯**: 如：GPU 整体健康状态。
- **交互**: 点击指标卡片可跳转到对应功能模块。

### 1. 数据管理与预处理
- **文件上传**:
  - 支持上传 .csv, .xlsx 等格式数据集。
  - 限制文件大小（例如 100MB）。
  - 类型校验、上传进度显示及错误反馈。
- **数据清洗**:
  - 提供非破坏性工具（如去重、缺失值处理）。
  - 允许用户预览清洗效果。
  - 生成独立的、可供训练使用的数据集。
- **格式转换**:
  - 自动化转换为 LLaMA-Factory 所需格式（如 Alpaca/ShareGPT JSON）。
  - 记录和展示警告或错误。
  - 生成详细数据集元信息 (dataset_info.json)，由后端动态生成和校验。
- **数据集管理**:
  - 列表展示、查询、选择、删除操作。
  - 版本化管理。
- **页面详细设计**:
  - **布局**: 表格视图为主，包含操作按钮。
  - **文件上传组件**:
    - 显眼的“上传数据集”按钮。
    - 支持拖拽上传区域。
    - 文件类型和大小限制提示。
    - 上传进度条和状态显示（成功/失败/进行中）。
    - 失败时显示具体错误信息。
  - **数据集列表**:
    - 表格展示：数据集名称、格式、大小、创建时间、状态（如：Ready, Processing, Error）。
    - 操作列：
      - “查看详情/预览”按钮：可点击查看数据集元信息、部分数据样本。
      - “格式转换”按钮：触发数据格式转换流程。
      - “删除”按钮（带确认弹窗）。
  - **数据清洗模块**:
    - 触发清洗的入口。
    - 清洗规则选择器（下拉/多选）。
    - 预览按钮。
    - 执行清洗按钮。

### 2. 模型训练任务编排
- **任务配置**:
  - Web 界面选择基础模型、配置 LLaMA-Factory 参数（如 LoRA 配置、学习率、Epochs、Batch Size）。
  - 提供默认值和解释性提示。
  - 输入校验（学习率正数、Epoch 整数等）。
- **资源预估与校验**:
  - 估算 GPU 显存需求，标记为“近似值”或“建议值”。
  - 考虑基座模型大小 + LoRA 参数 + 优化器状态。
  - 显存不足时警告并禁止启动，或提供“强制启动”选项需用户确认。
- **任务调度与管理**:
  - 支持多任务提交与排队，按顺序执行。
  - 实时更新任务状态（Pending, Running, Success, Failed, Cancelled）。
  - 防止重复提交相同任务。
  - 支持安全中止训练任务：找到 PID、发送 SIGTERM、清理资源（包括显存）。
- **模型保存与注册**:
  - **模型保存**: 训练成功后，系统必须将最终的模型权重（如 LoRA 权重）保存到持久化存储中。
  - **检查点支持**: 系统应提供按指定频率（如每 N 个 Epoch）保存训练检查点（Checkpoint）的选项，以支持断点续训。
  - **模型注册**: 每个成功训练并保存的模型，都必须被注册到系统中。注册信息（元数据）应存入数据库，至少包含模型名称、所基于的基础模型、训练参数、权重文件路径等，以实现版本控制和可追溯性。
- **页面详细设计**:
  - **布局**: 表单为主，结合任务列表。
  - **训练任务配置表单**:
    - 基础模型选择: 下拉框列出可用的预训练模型。
    - 数据集选择: 下拉框/多选框列出已准备好的数据集。
    - 训练参数配置 (参考 LLaMA-Factory 原生表单):
      - Epochs, Batch Size, Learning Rate, Optimizer (下拉)。
      - LoRA 配置: Rank (r), Alpha (lora_alpha), Dropout (lora_dropout)。
      - 显存预估结果显示区域（只读，附带警告）。
      - 其他 LLaMA-Factory 支持的参数。
    - 输入校验: 各输入框应有实时校验提示。
    - “启动训练”按钮: 触发任务提交，并附带显存警告确认。
  - **训练任务列表**:
    - 表格展示：任务 ID, 所选模型, 所选数据集, 状态, 起始时间, 预计/已用时长。
    - 操作列：
      - “查看日志”按钮：跳转到任务详情/日志页面。
      - “中止任务”按钮（带确认弹窗）。
      - “详情”按钮：展示任务配置和最终结果。

### 3. 实时监控 (任务监控与日志)
- **日志流**:
  - 实时捕获 LLaMA-Factory 的 stdout 和 stderr，通过 WebSocket 或 SSE 推送。
  - 前端以类终端形式展示。
- **指标流**:
  - 实时解析关键指标（如 Loss），以动态图表展示。
  - 提供进度条、当前 Epoch 和 Step 数。
- **硬件监控**:
  - 实时显示 GPU 显存占用、利用率、温度、功耗等。
- **页面详细设计**:
  - **布局**: 组合视图，包含日志区、图表区、硬件监控区。
  - **日志输出区**:
    - 类终端（Terminal-like）文本展示区域。
    - 实时滚动显示日志。
    - 支持日志过滤/搜索（可选）。
  - **指标图表区**:
    - 实时动态更新的折线图，展示 Loss、Accuracy 等指标。
    - 支持图例切换和缩放（可选）。
  - **硬件监控区**:
    - 显示当前可用 GPU 列表。
    - 每个 GPU 显示：ID, 型号, 显存使用率 (Bar Chart/Percentage), 温度, 利用率。
    - 高亮显示当前正在被训练任务使用的 GPU。

### 4. 模型推理与服务
- **模型加载**:
  - 后台加载预训练模型和 LoRA 权重，显示进度。
  - 展示已加载模型名称或来源。
- **对话接口**:
  - 交互式聊天界面，支持多轮对话。
  - 流式输出模型回复。
- **页面详细设计**:
  - **布局**: 聊天界面。
  - **UI 元素**:
    - 模型选择: 下拉框选择已完成训练的模型（或基础模型+LoRA 组合）。
    - 对话区域:
      - 左侧用户消息，右侧 AI 回复。
      - AI 回复需支持流式输出，显示逐字生成效果。
      - 显示 AI 思考中/正在生成状态。
    - 输入框: 用户输入消息的文本区域。
    - “发送”按钮: 提交用户消息。
    - “清空对话”按钮: 开始新的对话。
    - 推理参数配置 (可选): temperature, top_p, max_new_tokens 等参数的滑块或输入框。

### 5. 全链路可观测性
- **硬件状态**: 持续监控和展示 GPU 运行状态（如显存占用、利用率、温度、功耗等）。
- **任务状态**: 展示任务生命周期。
- **日志与指标**: 提供详尽训练日志和指标，实时低延迟传输。

## 非功能需求
- **安全性**: 防范常见漏洞，确保数据和模型安全。输入数据需做适当校验。
- **性能**: 轻量级设计，支持本地/私有云部署。页面加载速度快，交互响应迅速。
- **可观测性**: 全链路监控，确保低延迟实时反馈。
- **错误处理**: 全面异常捕获、日志记录、用户友好错误反馈。
- **边界情况**: 处理各种异常场景，如显存不足、格式错误等。
- **可靠性**: 页面稳定，不易崩溃。
- **可维护性**: 代码结构清晰，易于理解和修改。
- **兼容性**: 兼容主流现代浏览器（Chrome, Firefox, Edge）。

## 约束与假设
- 部署环境: 本地或私有云，支持 GPU。
- 依赖: 确保 LLaMA-Factory 兼容性，参考官方文档[1]。
- 其他: 所有实现必须完整、健壮，符合生产标准。

## 数据库设计
- **users 表**: 存储用户信息（ID, 用户名, 密码哈希等）。
- **datasets 表**: 存储数据集信息（ID, 用户ID, 文件名, 存储路径, 格式, 上传时间, 状态等）。
- **training_tasks 表**: 存储训练任务信息（ID, 用户ID, 数据集ID, 模型ID, 任务参数JSON, 状态, 开始时间, 结束时间, 错误信息等）。
- **models 表**: 存储可用基础模型信息（ID, 模型名称, 模型路径, 参数量等）。

[1]: LLaMA Factory 官方文档 (https://llamafactory.readthedocs.io/en/latest/)
