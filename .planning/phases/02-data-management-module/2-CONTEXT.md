# 阶段 2 上下文: 数据管理模块

本文件概述了作为第二阶段核心的数据管理模块的关键架构与实现决策。这些决策建立在第一阶段所搭建的基础设施之上。

## 1. 文件上传机制：分块与异步合并

- **决策**: 所有数据集上传将通过**分块（多部分）上传**机制处理。
- **理由**: 此方法对于确保大文件（根据需求高达100MB）的可靠上传至关重要。它通过（允许恢复失败的分块）提供了对网络中断的弹性，防止服务器超时，并通过精确的进度跟踪提供更好的用户体验。
- **工作流**:
  1.  **初始化**: 前端在开始上传前向后端请求一个 `uploadId`。
  2.  **上传分块**: 前端将文件切片，并并发地将分块上传至一个专用端点，每个分块都与该 `uploadId` 关联。
  3.  **完成上传**: 所有分块上传完毕后，前端发送一个最终请求，通知后端合并文件。
  4.  **后端合并**: 后端验证所有分块均已存在，将它们合并成最终的数据集文件，然后在 `datasets` 数据库表中创建相应的条目。

## 2. 数据处理：使用 Celery 进行异步任务执行

- **决策**: 所有长时间运行的数据处理任务，特别是**数据清洗**和**格式转换**，将由 Celery workers **异步**执行。
- **理由**: 同步执行这些可能耗时的任务会阻塞 FastAPI 服务器并导致 HTTP 超时，从而造成糟糕且无响应的用户体验。异步架构将初始的用户请求与繁重负载的实际执行解耦。
- **工作流**:
  1.  **API 触发**: 用户通过前端操作发起一个处理流程（例如，“转换为 Alpaca 格式”）。
  2.  **任务分派**: FastAPI 端点接收到请求，立即向 Celery 队列（通过 Redis）分派一个相应的任务，并即刻向前端返回一个 `202 Accepted` 响应及一个唯一的 `taskId`。
  3.  **前端监控**: UI 更新以显示“处理中”状态。然后，它使用 `taskId` 通过 WebSocket 订阅或轮询状态端点来监控任务进度。
  4.  **Worker 执行**: Celery worker 从队列中获取任务，执行数据转换，并在完成或失败时更新数据库中的任务状态。

## 3. API 端点设计 (RESTful)

- **决策**: 将实现一套 RESTful API 端点来编排数据管理的生命周期。这确保了前后端之间有一个清晰、无状态且可预测的接口。
- **核心端点**:
  - `POST /api/v1/datasets/initiate-upload`: 用于开始分块上传流程。
  - `POST /api/v1/datasets/upload-chunk`: 用于上传单个文件分块。
  - `POST /api/v1/datasets/complete-upload`: 用于完成上传并触发文件合并。
  - `GET /api/v1/datasets`: 用于列出用户所有可用的数据集。
  - `POST /api/v1/datasets/{dataset_id}/process`: 用于触发一个异步数据处理作业（清洗、转换）。
  - `DELETE /api/v1/datasets/{dataset_id}`: 用于删除一个数据集及其关联的文件。

这些决策共同确保了数据管理模块是健壮、可扩展的，并能提供与项目企业级目标一致的响应式用户体验。
