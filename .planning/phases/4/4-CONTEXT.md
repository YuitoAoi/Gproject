# 阶段 4 上下文: 实时监控

本文件概述了作为第四阶段核心的实时监控模块的关键架构与实现决策。

## 1. 数据传输架构：基于 WebSocket 和 Redis Pub/Sub 的推送模式

- **决策**: 所有实时监控数据（日志、指标、硬件状态）的传输将采用基于 **WebSocket** 的**推送 (Push)** 模式，而不是前端轮询的拉取 (Pull) 模式。
- **理由**: 此方案能提供真正的低延迟、实时数据更新，同时极大地降低了不必要的网络请求和服务器负载，架构清晰且可扩展性强，完全符合项目的实时性要求。
- **核心组件**: **Redis Pub/Sub** 将作为连接数据生产者（Celery worker）和数据消费者（WebSocket 服务）的中央消息总线。

## 2. 实时数据捕获机制

- **决策**: 针对不同类型的数据，采用不同的、专门的捕获机制，以实现与 LLaMA-Factory 训练过程的深度集成和解耦。
- **实现细节**:
  1.  **日志 (Logs) 捕获**:
      - **方式**: 实现一个自定义的 Python `logging.Handler`。
      - **逻辑**: 这个 Handler 会拦截所有由训练过程产生的日志记录，将其格式化，然后直接发布到与特定任务关联的 Redis 频道（例如，`log-channel:{task_id}`）。
  2.  **性能指标 (Metrics) 捕获**:
      - **方式**: 实现一个自定义的 Hugging Face `TrainerCallback`。
      - **逻辑**: 在训练生命周期的关键节点（如 `on_log`, `on_step_end`），该回调函数会被触发。它会提取如 Loss、学习率等关键指标，并将这些结构化的数据（JSON）发布到专用的 Redis 频道（例如，`metrics-channel:{task_id}`）。
  3.  **硬件状态 (GPU Stats) 捕获**:
      - **方式**: 在 Celery worker 内部，为每个正在运行的训练任务启动一个并行的**守护线程**。
      - **逻辑**: 该守护线程在一个独立的循环中运行，以固定的时间间隔（例如，每 2 秒）使用 `pynvml` 库查询相关 GPU 的实时状态（显存占用、温度、利用率等），然后将这些数据发布到其专用的 Redis 频道（例如，`gpu-stats-channel:{task_id}`）。

## 3. 端到端数据流 (End-to-End Data Flow)

- **数据生产 (Producers)**:
  - Celery worker 在执行训练任务时，会同时激活上述三种数据捕获机制，成为一个多源数据生产者，持续向 Redis 的多个频道发布数据。
- **数据消费与推送 (Consumer & Pusher)**:
  - **订阅请求**: 当用户访问某个任务的监控页面时，前端通过 WebSocket 连接向后端发送一个订阅请求（例如，`{"action": "subscribe", "taskId": "..."}`）。
  - **后端订阅**: 后端的 WebSocket 服务收到请求后，会代表该 WebSocket 连接去订阅 Redis 中与该 `taskId` 相关的所有频道。
  - **实时转发**: 一旦 WebSocket 服务从其订阅的任一 Redis 频道收到消息，它会立即将该消息原封不动地通过对应的 WebSocket 连接转发给前端浏览器。

这些决策共同构成了一个高效、实时、可扩展且职责清晰的监控系统架构，能够为用户提供卓越的“驾驶舱”体验。

## 4. 频道管理与系统弹性

- **决策**: 引入更严格的频道管理和全面的错误处理机制，以增强系统的健壮性和可维护性。
- **实现细节**:
  1.  **频道名称管理**:
      - **方式**: 在后端创建一个中央常量模块（例如，`app/core/constants.py`）。
      - **逻辑**: 所有 Redis 频道的名称前缀（如 `log-channel:`）将在此文件中作为常量进行定义。所有生产者和消费者都必须引用这些常量，而不是使用硬编码的字符串。这确保了全局的一致性，并简化了未来的维护。
  2.  **全局系统频道**:
      - **方式**: 引入一个名为 `system-notifications` 的全局 Redis Pub/Sub 频道。
      - **逻辑**: 此频道将用于广播非任务特定的系统级事件，例如 Celery worker 的上线/下线、全局系统告警或管理员通知。前端可以有一个全局的 WebSocket 处理器来订阅和响应该频道的消息。
  3.  **错误处理与弹性设计**:
      - **Redis 连接**: 所有与 Redis 交互的组件（Celery worker, WebSocket 服务）在进行连接时，都必须实现一个带有指数退避（Exponential Backoff）算法的自动重连循环。如果多次尝试后仍然失败，必须记录严重错误日志并触发告警。
      - **WebSocket 服务**: 必须优雅地处理客户端的意外断连（例如，关闭浏览器标签页），并确保清理该连接在后端的所有 Redis 订阅，以防止资源泄露。
