# 阶段 5 上下文: 模型推理与服务

本文件概述了作为第五阶段核心的模型推理与服务模块的关键架构与实现决策。

## 1. 模型生命周期管理：“异步预加载 + 主动通知 + 显存管理”模式

- **决策**: 采用一种兼顾用户体验和资源效率的综合模型管理模式。
- **理由**: 直接加载模型会导致长时间的等待，破坏交互体验；而让模型常驻内存则会浪费宝贵的 GPU 资源。本方案通过将加载过程异步化并赋予用户资源管理权，解决了这一矛盾。
- **工作流**:
  1.  **用户发起激活**: 在一个专门的模型管理界面，用户对一个已训练好的模型点击“激活”。
  2.  **后端异步加载**: 后端接收请求，立即将一个“加载模型至 GPU”的作业分派给一个专用的 Celery worker。API 立即返回，前端将模型状态更新为“正在激活...”。
  3.  **加载完成与主动通知**: Celery worker 加载成功后，更新数据库中该模型的状态为“已激活”，并通过全局的 Redis Pub/Sub 系统通知频道（`system-notifications`）发布一条“模型已激活”的消息。
  4.  **前端响应通知**: 所有客户端都监听该系统频道。收到激活成功消息后，前端会弹出一个全局通知（如 Toast），告知用户该模型已准备就绪。
  5.  **即时推理**: 用户可在聊天界面选择任何“已激活”的模型，并立即开始对话，无需等待加载。
  6.  **显存管理**: 
      - **手动停用**: 为每个“已激活”的模型提供“停用”按钮，用户点击后，后端将模型从 GPU 显存中安全卸载。
      - **自动超时 (可选)**: 可以引入一个策略，即如果一个已激活的模型在 N 小时内无任何推理活动，系统将自动将其停用以回收资源。

## 2. 对话交互实现：流式输出 (Streaming)

- **决策**: 所有的模型对话回复都必须以**流式输出**的方式呈现，以提供类似 ChatGPT 的逐字生成效果。
- **理由**: 流式输出极大地提升了人机对话的交互感和“智能感”，是现代 LLM 应用的标配。
- **技术方案**:
  1.  **后端 (Backend)**:
      - **推理函数**: 后端的推理服务将调用一个支持生成器 (generator) 模式的 LLM 推理接口。该函数会 `yield`（产出）模型生成的每一个新的 token。
      - **API 端点**: 创建一个专门的 FastAPI 端点（例如 `POST /api/v1/inference/chat-stream`），该端点将上述生成器包装在一个 `StreamingResponse` 对象中返回。
  2.  **前端 (Frontend)**:
      - **数据获取**: 前端将使用现代的 `Fetch API` 来调用此流式端点。
      - **数据处理**: 通过 `response.body.getReader()` 获取一个 `ReadableStream` 读取器。在一个循环中持续读取数据流中的新数据块（即新的 tokens），并实时地将它们追加到聊天界面的显示区域。

这些决策共同确保了模型推理模块将为用户提供一个低延迟、高交互性且资源可控的“模型游乐场”体验。
