# 阶段 5 计划：模型推理与服务

本计划概述了实现模型推理与服务模块的可执行步骤。计划基于 `5-CONTEXT.md` 中定义的架构决策。

## 第 1 部分：后端实现

### 任务 1.1：数据库模型扩展
- **动作**: 修改 `trained_models` 的 SQLAlchemy 模型，为其增加一个 `status` 字段（类型为字符串，例如：`unloaded`, `loading`, `active`）和一个 `last_used_at` 字段（类型为时间戳）。
- **动作**: 更新数据库迁移脚本以应用此更改。
- **验证**: `trained_models` 表结构已成功更新，包含了 `status` 和 `last_used_at` 列。

### 任务 1.2：模型加载/卸载服务
- **动作**: 在 `backend/app/services/` 中创建一个 `inference_service.py` 文件。
- **动作**: 在此文件中，创建一个 Celery 任务 `load_model_to_gpu(model_id: int)`。该任务负责：
  1.  将数据库中对应模型的 `status` 更新为 `loading`。
  2.  从磁盘加载模型权重到 GPU。
  3.  加载成功后，将模型对象保存在一个全局的、进程安全的字典或缓存中（键为 `model_id`），并将数据库中的 `status` 更新为 `active`。
  4.  通过 Redis Pub/Sub 的 `system-notifications` 频道发布“模型已激活”的通知。
- **动作**: 在 `inference_service.py` 中，创建一个函数 `unload_model_from_gpu(model_id: int)`。该函数负责从全局缓存中移除模型对象，执行必要的 GPU 显存清理，并将数据库中的 `status` 更新为 `unloaded`。
- **验证**: 单元测试可以表明，调用 Celery 任务能成功加载模型并更新状态。调用卸载函数能释放显存并更新状态。

### 任务 1.3：模型管理 API 端点
- **动作**: 在 `backend/app/api/` 中创建一个新的路由文件，用于模型推理管理。
- **动作**: 实现 `POST /api/v1/inference/models/{model_id}/activate` 端点。该端点分派 `load_model_to_gpu` Celery 任务。
- **动作**: 实现 `POST /api/v1/inference/models/{model_id}/deactivate` 端点。该端点调用 `unload_model_from_gpu` 函数。
- **动作**: 实现 `GET /api/v1/inference/models` 端点，用于获取所有已训练模型的列表及其当前激活状态。
- **验证**: 使用 API 测试工具可以成功调用这些端点，并观察到数据库中的模型状态按预期变化。

### 任务 1.4：流式聊天 API 端点
- **动作**: 在 `inference_service.py` 中，创建一个生成器函数 `stream_chat(model_id: int, prompt: str)`。该函数：
  1.  从全局缓存中获取已加载的模型。
  2.  调用模型的流式生成接口。
  3.  `yield` 模型生成的每一个 token。
- **动作**: 实现 `POST /api/v1/inference/chat-stream` 端点。该端点接收 `model_id` 和 `prompt`，然后调用 `stream_chat` 生成器并将其包装在 FastAPI 的 `StreamingResponse` 中返回。
- **验证**: 使用 `curl` 或一个简单的 Python 脚本调用此端点，可以观察到响应是以流式、逐块的方式返回的。

## 第 2 部分：前端实现

### 任务 2.1：模型管理页面
- **动作**: 创建一个新的页面和路由 `/models`，用于管理所有已训练的模型。
- **动作**: 在此页面上，使用 `el-table` 展示从 `GET /api/v1/inference/models` 获取的模型列表。
- **动作**: 表格应包含模型的名称、状态，以及一个“操作”列。根据模型状态，操作列显示“激活”或“停用”按钮。
- **动作**: 为“激活”和“停用”按钮绑定相应的 API 调用，并在请求发出后更新 UI 状态（例如，显示加载指示器）。
- **验证**: 页面能正确显示模型列表及其状态。点击按钮可以成功发送请求，并且 UI 状态会相应更新。

### 任务 2.2：全局通知处理器
- **动作**: 扩展 `frontend/src/api/websocket.js`（或创建一个新的全局服务），使其能够连接到一个用于接收系统级通知的 WebSocket 端点。
- **动作**: 当收到“模型已激活”的通知时，使用 Element Plus 的 `ElNotification` 服务弹出一个全局的、可点击的通知。
- **验证**: 在后端成功加载模型后，前端会弹出一个内容正确的通知。

### 任务 2.3：聊天界面
- **动作**: 创建一个新的页面和路由 `/inference/chat`。
- **动作**: 在此页面上，创建一个 `ChatInterface.vue` 组件，包含模型选择下拉框、聊天记录显示区和用户输入框。
- **动作**: 模型选择下拉框应只显示状态为 `active` 的模型。
- **动作**: 当用户发送消息时，使用 `Fetch API` 调用后端的 `/api/v1/inference/chat-stream` 端点。
- **动作**: 实现逻辑以读取 `ReadableStream`，并将接收到的 token 实时追加到聊天记录显示区中，以实现打字机效果。
- **验证**: 在聊天界面，可以选择一个已激活的模型并进行对话。模型的回复是以流畅的、逐字生成的方式显示的。

## 第 3 部分：最终验证 (UAT)

### 任务 3.1：端到端模型推理流程测试
- **动作**: 执行一个完整的用户流程：
  1.  导航到“模型管理”页面，选择一个“未加载”的模型，点击“激活”。
  2.  观察到模型状态变为“正在加载...”，然后变为“已激活”，同时收到一个全局通知。
  3.  导航到“聊天”页面，在下拉框中选择刚刚激活的模型。
  4.  发送一条消息，并观察到 AI 以流式的方式进行回复。
  5.  返回“模型管理”页面，点击“停用”按钮，观察到模型状态变回“未加载”。
- **验证**: 整个流程可以无差错地完成。所有状态变更都按预期在 UI 和数据库中正确反映。聊天体验流畅。
