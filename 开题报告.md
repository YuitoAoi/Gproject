基于LLama-Factory的大语言模型通用预训练平台设计
开题报告
1.	选题背景和意义
1.1	背景分析
近年来，深度学习与自然语言处理技术的飞速演进使大语言模型（LLMs）成为人工智能领域的重要研究方向[1][2]。这类模型通过在海量语料数据上进行预训练，可以学习丰富的语言知识和语义表示能力，在文本生成、机器翻译、问答系统、智能客服以及代码生成等多种应用场景中表现出较强性能。以ChatGPT、Qwen、LLaMA等模型为代表的大语言模型，展现出了强大的自然语言理解与生成能力。随着开源生态的蓬勃发展，越来越多的企业、高校及个人开发者希望在本地私有化环境中部署大模型，并通过输入自己的垂直领域数据（如医疗问答、企业知识库、客服话术）对模型进行微调，以满足特定场景的需求[5][6][7]。
然而，大语言模型的微调过程通常涉及复杂的环境配置、严苛的数据格式要求以及晦涩的命令行操作，对初学者和普通软件开发者而言具有较高的技术门槛。为简化这一流程，开源社区推出了LLaMA-Factory等优秀的微调框架，支持LoRA、QLoRA等参数高效微调算法[3][4]。但LLaMA-Factory在实际应用中仍主要依托命令行或简单的基础页面，存在以下痛点：
1.数据处理繁琐
框架仅接受特定格式的JSON数据集[9]，缺乏对日常业务数据（如Excel、CSV）的自动化解析能力。
2.训练过程犹如“黑盒”
命令行界面无法直观展示底层显卡的运行状态（如显存占用、温度），且对模型训练的Loss等关键指标缺乏实时的可视化监控，极易因参数配置不当导致显存溢出。
3.缺乏闭环管理
市面上如阿里云“百炼”等企业级平台虽然拥有完善的可视化工作流，但高度依赖云原生集群，无法直接应用于本地单机或小规模实验室算力环境[10]。
1.2	研究意义
针对上述痛点，本研究旨在利用现代软件工程思想，将底层复杂的算法训练流程封装为可视化Web软件产品。
在工程应用维度上，本系统计划设计一个“开箱即用”的轻量级大模型私有化微调工作站。通过提供“数据上传转换➡可视化参数配置➡异步训练调度➡实时大屏监控➡沙盒对话测试”的端到端工作流，大幅降低本地大模型微调的工程门槛。

在个人实践与学术维度上，本课题不再局限于简单的增删改查系统，而是深入探讨如何解决“长耗时AI任务与Web请求超时的矛盾”、“软硬件状态的全双工通信”等复杂工程问题，为计算机专业学生向LLMOps方向转型提供了一次具有前瞻性的实践。
2.	研究内容与功能概述
2.1	研究内容
本课题的总体目标是基于B/S架构，开发一款面向本地算力环境的大模型微调与管理平台。平台将对LLaMA-Factory的核心指令进行高度抽象与API封装，剥离算法运行与用户交互的耦合。系统开发完成后，用户只需通过浏览器，即可完成业务数据集的清洗录入、训练超参数的低代码配置、后台训练任务的异步调度，并通过Web界面实时监控GPU硬件状态与模型学习曲线，最终实现微调后模型的一键推理测试。
2.2	功能模块
本系统围绕大模型微调的生命周期，设计了四个核心功能模块。
1.数据集智能管理模块
解决异构数据转化的痛点。支持用户上传Excel、CSV格式的原始问答数据；系统后端使用Pandas自动进行数据清洗，并动态映射为大模型标准的ShareGPT/Alpaca JSON格式，最后自动改写底层配置完成数据集注册。
2.可视化训练调度模块
提供表单化的参数配置界面（支持新手推荐模式与专家自定义模式）。在下发任务前，系统根据基础模型参数量（7B/14B）和Batch Size进行初步显存预估；确认无误后，将任务推入后台队列异步执行。
3.全链路实时监控模块
系统的亮点功能。在训练期间不仅能够流式推送到前端控制台的训练日志，还能实时展示服务器底层的GPU显存占用率，并利用图表组件实时绘制损失下降曲线，打破训练黑盒。
4.模型沙盒推理测试模块
微调完成后，用户可在界面一键拉起推理服务。系统提供类似ChatGPT的对话沙盒，用户可通过文本交互直观检验模型的微调效果。

2.3	非功能性要求
1.性能与响应要求
普通管理页面的API接口响应时间需控制在500ms以内；模型推理沙盒对话的流式输出首字响应时间应控制在2秒以内。
2.并发与调度要求
由于模型微调属于极长耗时的重度计算任务，系统必须具备异步任务排队与解耦能力，保证后台训练的满负荷运行绝对不能阻塞前端Web服务的正常访问请求。
3.安全性要求
保证本地私有化部署的特性，上传的数据不出内网；对用户上传的文件进行严格的格式和大小校验，防止系统路径穿越和恶意命令注入攻击。
4.可视化要求
前端UI界面需符合现代中后台系统规范，图表组件及硬件监控仪表盘需保证高帧率平滑渲染，数据更新无明显闪烁或卡顿
2.4	拟解决的关键问题
1.异构数据到算法输入标准的自动化对齐问题
不同业务部门提供的数据格式千奇百怪，如何编写鲁棒的转化算法，提取出问答对齐并自动生成合规的dataset_info.json。
2.长耗时 AI 计算与 Web 响应超时的矛盾
传统的HTTP请求/响应模型无法挂起长达数小时的训练脚本，如何建立稳定的后台任务队列和子进程管理机制。
3.系统软件层与底层算法、硬件层的实时双向通讯
LLaMA-Factory运行在终端环境，如何用Web后端代码非侵入式地捕获其标准输出流（stdout）、读取训练状态文件，并跨越协议实时推送到前端浏览器。
3.	技术路线与方案
3.1	技术路线选型
系统采用前后端分离的现代化架构，紧密结合轻量级分布式任务队列与本地算法基座，以实现最佳的开发与运行效率。
开发语言
Python3.10+与JavaScript。使用Python方便直接调用数据处理库及算法框架API。
前端框架
采用Vue 3搭配Vite构建工具，结合Element Plus组件库快速搭建首页、模型管理页、训练任务页；引入ECharts进行训练曲线及硬件仪表盘的高性能渲染。
后端框架
采用FastAPI + Celery框架，FastAPI + WebSocket实现前后端实时通信，Celery实现异步任务调度管理大模型训练进程。
数据库中间件
采用Redis作为缓存与消息队列，用于支撑任务队列和高频硬件状态的转发。采用MySQL 8.0存储用户信息、数据集信息及训练任务记录；
底层算法
采用LLaMA-Factory，兼容主流的Qwen、Llama3等开源模型，利用其底层封装的transformers和peft库进行LoRA微调。此外，本系统预留了对接测试用例生成的接口扩展能力[8]。
开发与部署环境
前端及后端业务代码在Windows环境开发；考虑到GPU算力调用，模型训练及最终的系统部署环境需搭建在Linux(Ubuntu) + CUDA 12.x + PyTorch 2.x的云端或本地算力服务器上。

3.2	关键技术实现
1.基于Celery的异步任务调度与子进程管理技术
针对长耗时任务，系统引入Celery分布式任务队列。前端下发训练请求后，FastAPI仅验证参数并将任务信息组装成JSON写入MySQL，随后推入Redis队列并立即返回前端“任务已受理”。后端的Celery Worker监听队列，取出任务后，利用Python的subprocess.Popen模块动态拼接LLaMA-Factory的启动命令（llamafactory-cli train config.yaml）并以子进程方式运行，通过进程隔离解决了web后端直接调用Llama factory API而导致阻塞的问题
2.基于WebSocket的全双工实时监控技术
为打破训练“黑盒”，后端整合Nvidia官方提供的pynvml库，设计定时器每秒直接读取底层显卡驱动的真实显存与温度数据；同时，Celery Worker接收到subprocess的日志输出后，会进行解析，实时监测LLaMA-Factory训练时在输出目录中生成的trainer_state.json文件，提取最新的Loss值和Step进度。这些异构数据被统一打包，通过FastAPI建立的WebSocket全双工通道实时推送至Vue前端，再经由ECharts的setOption方法实现动态增量渲染。
3异构数据集自动化清洗技术
针对数据集痛点，使用pandas库对用户上传的.xlsx等文件进行读取，清除空行和异常字符。随后遍历数据，将对应的“问题”与“答案”列重新组织为包含{"instruction": "", "input": "", "output": ""}键值的字典列表，写入服务器本地JSON文件，并利用文件读写机制自动修改算法框架配置完成数据集挂载，实现数据的零代码接入。
