### 仪表盘界面设计
页面可以划分为三大横向区域。  

**模块一（顶部区域）：全局状态概览**  
- 位置建议：页面最上方，横向一排卡片（数据看板，art-design-pro的界面中已有，可以使用他的“数字滚动统计卡片”）。
- 功能目标：提供系统资产和当前负荷的宏观感知。  
- 具体展示项：
    - 正在运行的任务队列 (运行中/排队中)：列表展示当前正在执行的模型训练、数据清洗、模型导出任务。包含简化的实时进度条（对接 WebSocket metrics-channel 的 Progress 数据）。
    - 已纳管数据集总数：展示当前系统内就绪的数据集数量和总容量（例如 12 个 / 4.5 GB）。
    - 已微调模型总数：展示产出的可用模型数量。  
    - 累计算力消耗： 展示机器总共运行了多少小时的训练任务，能给用户极强的“平台正在产生价值 (ROI)”的获得感。（例如：累计训练：128小时。副文本：本周新增：12小时）

### 任务调度中心
任务调度中心是全平台 AI 任务（数据预处理、预训练、微调、评估、导出）的“中央交通枢纽”。完全聚焦于业务流水线与任务生命周期管理。该页面不是针对某一个任务的深度调试，而是解决“当前系统在跑什么、下一个轮到谁、资源够不够、有没有任务饿死或失败”的宏观调度问题。  

**全局架构与设计共识:**  
1. 纯任务视角：页面上出现的每一个数字、每一条记录，描述对象都是“任务”.
2. 布局范式：顶部(任务大盘与逻辑资源看板) + 中间(任务控制台列表) + 右侧(任务级日志与指标抽屉)。
3. 视觉风格：沿用 art-design-pro 的白蓝主色调。调度系统的交互密度高，色彩仅用于状态标识和告警。

**模块功能与细节考量**
- 隔离性原则：这里的“终端日志”只显示 LLaMA-Factory 引擎输出的关于该任务本身的 Python/CUDA 运行日志（如 loss=0.8, grad_norm=1.2），绝对不混入“服务器 CPU 温度过高”这种底层基建告警。
- “所见即所得”的数据流转：
前端 Vue3 使用 ECharts。
后端 FastAPI 起后台线程，利用 LLaMA-Factory 训练过程持续输出的 trainer_state.json 文件。
通过 WebSocket 每 5 秒将最新的 {step, loss, learning_rate} 结构化数据推给前端，形成动态绘制的曲线，极大提升平台的“高级感”。
- 在 Vue3 的实现中，必须严格遵守以下解耦规范：
1. 组件级状态隔离：
    - 不要在顶层的父组件（如 Monitor.vue）里维护 const logs = ref([]) 和 const metrics = ref([])。
必须拆分成独立子组件：<LossChart />, <Terminal />, <CheckpointTable />。
2. 连接分离：
    - <Terminal /> 组件内部自己建立并维护 WebSocket A（专门接收日志文本）。
    - <LossChart /> 组件内部自己建立并维护 WebSocket B（专门接收 JSON 指标）。
    - 这样，当大量日志涌入时，Vue 的响应式系统只会触发 <Terminal /> 的局部更新，左侧的 ECharts 曲线和底部的参数列表完全不受影响，做到真正的“稳如老狗”。

**模块功能与细节考量**
- **左侧 (算子面板)**：采用折叠面板（Accordion）包裹不同的清洗逻辑。用户像搭积木一样开启/关闭算子。使用 Pinia 集中管理一个深层嵌套的 JSON Config 对象。

    - **模型字段映射**：该模块的作用是对数据格式进行清洗对齐。因为用户上传的原始表格（Excel/CSV）千奇百怪（表头可能是“提问”、“回答”、“Q”、“A”、“原文”、“译文”等），我们需要通过“映射（Mapping）”将它们强制归一化为 LLaMA-Factory 认识的标准三要素。以下是这三个核心字段的要求：
        - instruction：  
        含义：指令/任务说明  
        必填状态：必填  
        映射场景：如果用户的表格是一个问答对（Q&A），通常将 Question（提问）列映射给 instruction。
        - input：  
        含义：执行该指令所需要的“已知条件”或“补充材料”。
        必填状态：非必填（可为空）。在很多简单的问答中，指令本身已经包含了所有信息，此时 input 就是空的。
        映射场景：映射场景：当用户的语料是“阅读理解”或“文本翻译”时，通常将“原文”映射给 input，将“指令说明”映射给 instruction。
        - output：  
        含义：这是模型需要学习的“答案”（Ground Truth）。  
        必填状态：必填。  
        映射场景：通常映射为用户表格中的 Answer（回答）、Result（结果）、Translation（译文）列。  

        **对于表头的字段名下拉框应显示什么内容的问题**：必须动态读取文件内容来生成源表头，绝对不能只靠系统预设。如果下拉框里只有系统预设的字段，用户根本无法将他们文件里的 患者主诉 映射给大模型的 instruction。这个平台就变成了“残废”。  

        **如何实现“动态获取源表头”？**：我们在前面的需求文档中提到，进入 Step 2 时，前端会向 FastAPI 后端请求 前 100 条样本数据 用于右侧的沙盒预览。这 100 条样本，就是获取表头的“钥匙”。

        **技术实现流转图**：
        - 解析 (Backend)：FastAPI 读取用户 CSV 的第一行（或者 JSON 的顶级 Key），提取出所有的列名（例如：["患者主诉", "医生诊断", "就诊时间"]）。
        - 传输 (Network)：随着 100 条预览数据一起返回给 Vue3 前端。
        - 渲染 (Frontend)：Vue3 拿到这个数组后，将其渲染到左侧映射规则的下拉列表中。
        
        所以这个模块的设计逻辑应为:

        |请选择源数据表头|      |映射到模型字段|
        |:-------:|:----:|:------:|
        |[ 请选择 🔽]|→|[ instruction(必填)] |
        |[ 请选择 🔽]|→|[ input ] |
        |[ 请选择 🔽]|→|[ output(必填)] |

        用户点击下拉框，看到的是他自己文件里真实存在的列名，这种体验是极其顺畅的。

        **特殊情况处理**：如果没有表头怎么办？ (TXT 或纯文本)：
        如果用户上传的是一个 .txt 文件（比如一本《西游记》小说），这种文件是没有表头的。  
       **处理方案**：系统检测到非结构化文本（txt），后台在解析样本时，人为给它捏造一个虚拟列名，叫做 [文件原始内容]。
        对于这种纯文本预训练数据，映射界面会自动变成这样：
        |请选择源数据表头|      |映射到模型字段|
        |:-------:|:----:|:------:|
        |[ 文件原始内容 ]|→|[ text (预训练纯文本)] |

- **右侧 (沙盒面板)**：采用 Diff 思想。不仅要展示清洗后的结果，还要用红绿对比高亮展示“哪些字被动过”，彻底消除用户的黑盒恐惧。
- **💡 架构考量（前端性能与零延迟预览）**：
    - **坚决避免高频 API 调用**：在 Step 2 的实时预览中，当用户拨动“移除 HTML”或“手机号脱敏”开关时，不要每次都向 FastAPI 发起请求。网络延迟（哪怕是 200ms）都会摧毁“所见即所得”的快感。
    - **本地 JS 沙盒策略**：在进入 Step 2 时，Vue3 向后端请求并缓存 100 条 原始随机样本（JSON 格式）。对于正则替换、长度截断、HTML剥离等基础算子，前端利用 JS 引擎在浏览器内存中直接执行并渲染 Diff。
    - **优雅降级**：只有当用户开启了“MinHash 模糊去重”等无法在单条数据中体现、必须依赖全局计算的复杂 ML 算子时，前端沙盒仅展示一个占位提示：“全局去重效果将在完整任务执行后体现”。

**模块功能与细节考量**
- **左侧 (质量漏斗)**：将抽象的数据流转具象化为漏斗图 (Funnel Chart)。大模型工程师极其关注数据的“留存率”（Yield Rate）。清楚地告诉用户“损失的数据去哪了”（因为太短？还是因为重复？），是对算法透明度的最高体现。
- **右侧 (终端日志)**：模仿 Console 的 UI，给用户强烈的“系统正在硬核运算”的安全感和掌控感。
- **架构考量（长连接与状态保持）**：
    - **通信协议**：点击提交后，前端通过 HTTP POST 发送配置。后端将任务丢给 Celery/Redis，返回 task_id。前端立即建立 WebSocket (或 Server-Sent Events, SSE) 连接，订阅该 task_id 的实时日志和进度。
    - **非阻塞离线执行**：页面右下角必须提供 [返回资产池] 按钮。下方小字明确告知用户：清洗任务在服务器端运行，关闭浏览器或跳转页面不会导致任务中断。数据清洗任务也需要推入“数据集管理”页面的任务队列中。
    - **回溯机制**：如果用户跳转走后，在“数据集管理”页面点击“查看日志”，前端根据该数据集绑定的最新 task_id，重新渲染这个 Step 3 页面，并从 FastAPI 拉取历史日志（而非从头覆盖）。
